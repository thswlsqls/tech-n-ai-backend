블로그 시리즈: LangChain4j Agent Tool 무한 호출 디버깅기

  ---
  1편: "LLM Agent는 왜 같은 Tool을 30번 호출했을까"

  부제: LangChain4j 기반 Agent의 Tool Loop 문제 발생과 근본 원인 분석

  도입부

  - 프로덕션 환경에서 "RSS 피드 전체 수집해주세요" 한마디에 Agent가 폭주한 상황 소개
  - LangChain4j의 exceeded 30 sequential tool invocations 예외와 프론트엔드 연결 끊김(Broken pipe)

  본문 구성

  1. 두 가지 사고 사례

  ┌──────────┬─────────────────────────────────────────────────┬────────────────────────────────────────────┐
  │          │            Case 1: collect_rss_feeds            │       Case 2: fetch_github_releases        │
  ├──────────┼─────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 트리거   │ "RSS 피드 전체 수집해주세요"                    │ "OpenAI Python SDK 최신 릴리스             │
  │          │                                                 │ 조회해주세요"                              │
  ├──────────┼─────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 증상     │ 수집 성공 응답(new=0, dup=41)을 받고도 재호출   │ 동일 인자로 30회 연속 호출                 │
  │          │ 반복                                            │                                            │
  ├──────────┼─────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 소요     │ ~2분 30초                                       │ ~50초                                      │
  │ 시간     │                                                 │                                            │
  ├──────────┼─────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 결과     │ 30회 제한 도달 → 예외 → Broken pipe             │ 30회 제한 도달 → 예외                      │
  └──────────┴─────────────────────────────────────────────────┴────────────────────────────────────────────┘

  - 각 Case의 서버 로그 발췌와 호출 시퀀스 시각화
  - LLM(gpt-4o-mini)이 어떤 판단 흐름으로 재호출을 결정했는지 추론

  2. 근본 원인: 3계층 복합 원인

  - LLM 판단 계층: @Tool description에 종료 조건이 없어 LLM이 "작업 완료" 시점을 판단하지 못함. 성공
  응답(dup=41)을 "아직 할 일이 남았다"로 해석
  - 에러 처리 계층: "다른 방법을 시도해주세요" 메시지가 LLM에게 "같은 Tool을 다시 호출하라"는 신호로 작용
  - Adapter 계층: 외부 API 예외를 List.of()로 삼켜서 LLM이 에러와 "데이터 없음"을 구분 불가 → "이상하다, 다시
  해보자" 루프 유발

  3. LLM Agent 시스템에서 Tool 반환값의 의미

  - 전통적 함수 반환값 vs LLM Agent Tool 반환값의 차이
  - Tool의 모든 출력(성공 응답, 에러 메시지, 빈 결과)이 LLM의 다음 행동을 결정하는 신호라는 관점
  - 기존에 fetch_github_releases → collect_github_releases 간에는 방어가 있었지만, 다른 Tool에는 동일 패턴이
  적용되지 않았던 이유와 교훈

  4. LangChain4j의 Tool 호출 제한 메커니즘

  - maxSequentialToolInvocations(30) 기본값의 의미
  - 이 제한이 "안전장치"가 아닌 "마지막 방어선"인 이유 — 30회에 도달했다는 것 자체가 이미 비용 낭비

  마무리

  - 30회 제한에 의존하는 것은 방어가 아니라 실패의 증거
  - 2편에서 다룰 3계층 방어 패턴 예고

  ---
  2편: "LLM Agent의 Tool Loop를 막는 3계층 방어 패턴"

  부제: 멱등성 추적, 프롬프트 힌트, 에러 메시지 설계로 Agent 폭주 방지하기

  도입부

  - 1편에서 분석한 3계층 원인에 대응하는 3계층 방어 전략 개요
  - 방어 설계의 핵심 원칙: "LLM은 지시하지 않으면 멈추지 않는다"

  본문 구성

  1. 1계층 — Tool-level 멱등성 방어 (ToolExecutionMetrics)

  - ToolExecutionMetrics 클래스 설계: Tool 호출마다 "이미 처리된 인자"를 Set<String>으로 추적
  - 방어 흐름: Tool 호출 → 입력값 검증 → 중복 체크 → 실제 실행 → 완료 마킹
  - 중복 호출 시 BLOCKED 응답 반환 → 3회 초과 시 AgentLoopDetectedException으로 graceful 종료
  - 설계 포인트:
    - "전체 수집" 처리: provider="" → _ALL_ 키로 저장하여 이후 개별 provider 조회도 차단
    - 빈 결과도 마킹: 재시도해도 동일 결과이므로 빈 결과도 "완료"로 처리
    - 공유 카운터: fetchLoopBlockedCount를 fetch_*와 scrape_*가 공유하여 Tool 간 교차 루프 감지
  - 코드 변경 전후 비교 (collectRssFeeds, fetchGitHubReleases, scrapeWebPage)

  2. 2계층 — LLM 프롬프트 방어 (@Tool description)

  - @Tool annotation의 value 필드가 LLM의 Tool 선택과 사용 판단에 미치는 영향
  - 변경 전: 기능 설명만 존재 → 변경 후: 종료 조건과 재호출 금지 힌트 명시
  - 예시: "이미 수집한 provider를 다시 호출하지 마세요. 수집 완료 후 결과를 요약하고 작업을 종료하세요."
  - 프롬프트 방어의 한계: 소프트 방어이므로 LLM이 무시할 수 있음 → 1계층(하드 방어)과 반드시 병행

  3. 3계층 — 에러 메시지 & 예외 전파 방어

  - ToolErrorHandlers 에러 메시지 변경: "다른 방법을 시도해주세요" → "이 Tool을 동일한 인자로 재시도하지 마세요.
  해당 작업을 건너뛰고 다음 작업으로 진행하세요."
  - GitHubToolAdapter 예외 전파: catch → List.of() → catch → RuntimeException 재전파
  - 핵심 원칙: LLM Agent 시스템에서 예외를 삼키면 무한 루프의 직접적 원인이 된다

  4. 수정 전후 동작 비교

  ┌──────────────────┬──────────────────────────────┬───────────────────────────────────┐
  │                  │           수정 전            │              수정 후              │
  ├──────────────────┼──────────────────────────────┼───────────────────────────────────┤
  │ Case 1 (collect) │ 30회 실제 API 호출, 2분 30초 │ 4회 실제 수집 + 4회 BLOCKED, 25초 │
  ├──────────────────┼──────────────────────────────┼───────────────────────────────────┤
  │ Case 2 (fetch)   │ 30회 API 호출, 50초          │ 1회 실제 호출 + 4회 BLOCKED, 5초  │
  └──────────────────┴──────────────────────────────┴───────────────────────────────────┘

  - 전체 Tool별 방어 현황 매트릭스 (5개 Tool × 4가지 방어 메커니즘)

  5. 함께 발견된 부수 이슈들

  - MongoDB 중복 문서 Race Condition → find+save 대신 MongoTemplate.upsert atomic operation 적용
  - Agent 실패 시 ASSISTANT 메시지 미저장 → 성공/실패 무관하게 항상 저장으로 변경
  - 페이지네이션 size 불일치 → totalPageNumber 대신 totalSize 기반 재계산
  - 공통 교훈: Agent 시스템은 Tool 실행뿐 아니라 데이터 동기화, 이력 관리, 조회 정합성까지 함께 검증해야 함

  6. 재발 방지 체크리스트

  - 새로운 Tool 추가 시: ToolExecutionMetrics에 추적 필드 추가 + @Tool description에 종료 힌트 포함
  - 외부 API Adapter: 일반 예외를 삼키지 않고 명시적 전파
  - 에러 메시지: "재시도하지 마세요" 명시
  - 테스트: 중복 호출 차단 + AgentLoopDetectedException 발생 검증 테스트 필수 작성

  마무리 — 핵심 교훈 4가지

  1. LLM에게 주는 모든 텍스트는 행동 지시다 — 에러 메시지, 빈 응답, Tool description이 곧 LLM의 다음 행동
  2. Tool은 멱등성을 기본값으로 설계하라 — 같은 인자로 여러 번 호출해도 안전해야 한다
  3. 방어는 소프트(프롬프트) + 하드(로직) + 시스템(에러 핸들링) 다층으로 — 어느 한 계층만으로는 불충분           
  4. 예외를 삼키지 말라 — 전통적 시스템에서는 graceful degradation이지만, LLM Agent에서는 무한 루프의 원인
